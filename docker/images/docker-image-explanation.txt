we need to know how to build our own images, but first we need to know what is an image and why do we need it.
What is an image: 
- a read-only, static template that contains all the instructions and components needed to create a Docker container


docker image is read only, anytime we run a container, the container builds from the image and it only adds
an executable layer on top of what containers images already have. 

each instruction line in docker image is a layer. 

Here are the two main principles of docker image:
1. docker images are immutable. Once an image is created it can't be modified. You can only make a new image or add changes on top of it
2. Container images are composed of layers. Each layer represnt a set of file system changes that add, remove or modify files. 


we write docker images in Dockerfile. After that we need to run it to produce the docker images.
we do that using this command, "docker image build ."

then after that, just as we do with containers, we do this, docker image <command> <option>

When docker wants to build an image it only asks for two precise things,
- The docker file (Dockerfile)
- The context for the build

And that happens here:
- docker file - Dockerfile
- context - Where the docker file is located, that is done using the small dot (".")


//what about the tagging? (is this not --name in conatiner?):
- ealier on we said we can build our customer docker file by just writing "docker image build ."
- So the only way to identify this is by the ID only once the build is complete. 
- So how do we solve this? In contaienr we had an option we used called --name which allowed us to name our continaters we
starting them so as it is easier to identify them. 
- in images we use the --tag or -t 
-So it would be something like this:
`docker image build --tag <the name you want to give it> .`  DO NOT FORGET the dot (".") at the end. 

But there is still more to tag we can do, here is the "more" to do:
- Assume we want to call our image build a certain name but we also need to include the version, we can do it, here is how:
--tag <image repository>:<image tag> 

For example, if we want to run a verison of MySQL version 5.7 we do this:
docker container MySQL:5.7 - then this will give us the 5.7 verison.

We can do it here for out nginx project like this:
docker image build --tag custom-nginx:packaged . (NEVER FORGET THE .)


//Everything has an end, how about deleting the images?
- We use a simple thing here, we first list the images to check which one we need out
- we use `docker images` or `docker image ls`
- Now that we have them, we can see which one we dont need and remove it, here is how:
docker image rm <image identifier>
- You can also prune all images that are un-tagged and dangling. 
- Here is how, `docker image prune --force` (though we need to be careful with this one)
- You can also use the --all or -a to remove all the images in the cached registry in your machine. 
`docker image rm --all`
